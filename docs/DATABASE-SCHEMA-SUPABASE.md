## 1. Visi√≥n General

### Descripci√≥n del Sistema
E-commerce para Costa Rica con gesti√≥n completa de productos, pedidos, usuarios y notificaciones, utilizando Supabase como backend completo.

### Stack Tecnol√≥gico
- **Backend**: Supabase (PostgreSQL + Auth + Storage + Realtime)
- **Base de Datos**: PostgreSQL 15+ con Row-Level Security (RLS)
- **Autenticaci√≥n**: Supabase Auth (JWT-based)
- **Frontend**: React + TypeScript + Vite
- **API**: Supabase Client (auto-generado)

### Principios de Dise√±o
- ‚úÖ **Seguridad por Dise√±o**: RLS en todas las tablas sensibles
- ‚úÖ **Roles Separados**: Tabla `user_roles` independiente
- ‚úÖ **SECURITY DEFINER**: Funciones para evitar recursi√≥n RLS
- ‚úÖ **Normalizaci√≥n**: 3NF para evitar redundancia
- ‚úÖ **Performance**: √çndices en queries cr√≠ticos
- ‚úÖ **Realtime**: Suscripciones a cambios en tiempo real

### 5.1 Tabla: `public.users`

**Descripci√≥n**: Perfiles de usuarios (1:1 con `auth.users`)

**‚ö†Ô∏è CR√çTICO**: 
- NO almacenar roles aqu√≠ (usar `user_roles` table)
- Relaci√≥n 1:1 con `auth.users` usando mismo UUID

**SQL**:

```sql

CREATE TABLE public.users (
    id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    phone VARCHAR(20),
    avatar_url TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    
    CONSTRAINT valid_email CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}$'),
    CONSTRAINT valid_phone CHECK (phone ~* '^\+?[0-9\s\-]{8,20}$' OR phone IS NULL)
);

-- √çndices
CREATE INDEX idx_users_email ON public.users(email);

-- Comentarios
COMMENT ON TABLE public.users IS 'Perfiles de usuarios (1:1 con auth.users)';
COMMENT ON COLUMN public.users.id IS 'Mismo UUID que auth.users.id';
COMMENT ON COLUMN public.users.email IS 'Email sincronizado con auth.users';

-- Enable RLS
ALTER TABLE public.users ENABLE ROW LEVEL SECURITY;
```

## 6. Funciones Security Definer

### ‚ö†Ô∏è ¬øPor qu√© SECURITY DEFINER?

**Problema**: Recursi√≥n infinita en RLS policies

```sql
-- ‚ùå ESTO CAUSA RECURSI√ìN INFINITA:
CREATE POLICY "Admins ven todo" ON public.orders
FOR SELECT USING (
  (SELECT role FROM public.user_roles WHERE user_id = auth.uid()) = 'admin'
  --      ‚Üë Esta query consulta user_roles, que tambi√©n tiene RLS
  --        causando recursi√≥n infinita
);
```

**Soluci√≥n**: Funci√≥n SECURITY DEFINER

```sql
-- ‚úÖ CORRECTO: Funci√≥n SECURITY DEFINER evita recursi√≥n
CREATE OR REPLACE FUNCTION public.has_role(_user_id UUID, _role public.app_role)
RETURNS BOOLEAN
LANGUAGE SQL
STABLE
SECURITY DEFINER  -- ‚Üê Ejecuta con privilegios del due√±o, bypasea RLS
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1
    FROM public.user_roles
    WHERE user_id = _user_id
      AND role = _role
  )
$$;

-- Ahora la policy puede usar la funci√≥n sin recursi√≥n:
CREATE POLICY "Admins ven todo" ON public.orders
FOR SELECT USING (public.has_role(auth.uid(), 'admin'));
```

### Funciones Requeridas

```sql

-- Descripci√≥n: Verifica si un usuario tiene un rol espec√≠fico

RETURNS BOOLEAN

    SELECT 1
    FROM public.user_roles

$$;

-- Funci√≥n: is_order_owner()
-- Descripci√≥n: Verifica si el usuario es due√±o de una orden

    FROM public.orders
```

## 7. Triggers Autom√°ticos

### 7.1 Trigger: Crear perfil al registrarse

```sql
-- Funci√≥n: handle_new_user()
-- Descripci√≥n: Crea perfil en public.users cuando se registra un usuario
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  INSERT INTO public.users (id, name, email, phone)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data->>'name', 'Usuario'),
    NEW.email,
    NEW.raw_user_meta_data->>'phone'
  );
  RETURN NEW;
END;
$$;

-- Trigger: Ejecutar handle_new_user() despu√©s de INSERT en auth.users
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW
  EXECUTE FUNCTION public.handle_new_user();
```

## 8. Pol√≠ticas RLS (Row-Level Security)

### ‚ö†Ô∏è Principios de RLS

1. **Siempre habilitar RLS** en tablas con datos de usuarios
2. **Usar funciones SECURITY DEFINER** para evitar recursi√≥n
3. **Pol√≠ticas espec√≠ficas** por operaci√≥n (SELECT, INSERT, UPDATE, DELETE)
4. **Principio de menor privilegio**: Solo acceso necesario

```sql
      'product_image', oi.product_image,
      'quantity', oi.quantity,
      'price_at_time', oi.price_at_time,

GROUP BY o.id, u.id;
```
